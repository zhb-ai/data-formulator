### 修复：Excel 日期类型单元格导致 React 渲染崩溃 (error #31)

**问题现象**：上传包含日期列的 Excel 文件时，页面报错 `Minified React error #31; args=[object Date]`，整个应用崩溃。

**根本原因**：ExcelJS 解析 Excel 单元格时返回原生 JavaScript `Date` 对象，而 React 无法将 `Date` 对象作为子节点渲染。该问题在数据管道的多个层级均缺少防护。

**涉及文件与修复内容**：

| 文件 | 问题 | 修复方式 |
|------|------|----------|
| `src/data/utils.ts` | `loadBinaryDataWrapper` 中 `cell.value` 直接存储 Date/富文本/公式等 ExcelJS 复杂类型 | 新增 `resolveExcelCellValue()` 函数，统一将 Date 转 ISO 字符串、富文本合并为纯文本、公式取结果值、超链接取文本 |
| `src/data/types.ts` | `coerceDate()` 函数直接 `return v`，Date 对象透传不做任何转换 | 改为 `v instanceof Date ? v.toISOString() : v`，确保 Date 对象不会泄漏到渲染层 |
| `src/views/ReactTable.tsx` | 无 `format` 时直接渲染 `{value}`，Date/Object 类型触发 React error #31 | 增加 `typeof value === 'object'` 判断，对象类型自动 `String()` 转换 |
| `src/views/SelectableDataGrid.tsx` | 同上，无 `format` 时 `{cellValue}` 直接渲染 | 同上，增加对象类型安全渲染 |
| `src/views/DataLoadingThread.tsx` | `format: (v) => v` 原样返回，不做任何转换 | 改为对象类型自动 `String()` 转换 |

**防御深度**：修复采用多层防御策略：
1. **数据源层**（`resolveExcelCellValue`）：在 Excel 解析时就将复杂类型转为基本类型
2. **类型系统层**（`coerceDate`）：作为兜底，确保 Date 对象在类型推断后被转为字符串
3. **渲染层**（ReactTable / SelectableDataGrid / DataLoadingThread）：即使前两层遗漏，渲染时也不会因对象类型崩溃

### 修复：匿名/登录状态切换后 ChartRecBox 崩溃 (Cannot read properties of undefined reading 'model')

**问题现象**：从匿名分析 chart 对话时切换到登录状态，再次访问缓存数据，页面报错 `Cannot read properties of undefined (reading 'model')`，整个应用崩溃。

**根本原因**：
1. `dfSelectors.getActiveModel` 在 `globalModels` 和 `models` 都为空时返回 `undefined`（尽管类型标注为 `ModelConfig`）
2. `ChartRecBox` 组件直接访问 `activeModel.model` 而不做空值检查

状态切换场景：匿名状态的缓存中 `selectedModelId` 指向一个在登录后不存在的模型，且登录后模型列表可能尚未从服务端加载完成，导致 `getActiveModel` 返回 `undefined`。

**涉及文件与修复内容**：

| 文件 | 问题 | 修复方式 |
|------|------|----------|
| `src/app/dfSlice.tsx` | `getActiveModel` 返回类型标注为 `ModelConfig` 但实际可能返回 `undefined` | 修正类型为 `ModelConfig \| undefined` |
| `src/views/ChartRecBox.tsx` | 第 217 行直接访问 `activeModel.model` 无空值检查 | 使用 `activeModel?.model ?? ''` 安全访问 |

### 修复：模型不可用时无限重试测试连接

**问题现象**：当配置的模型 API Key 无效（如 401 错误）时，前端疯狂重复调用 `POST /api/agent/test-model`，导致大量无效请求。

**根本原因**：`DataFormulator.tsx` 中自动测试模型的 `useEffect` 存在死循环：

1. `dfSelectors.getAllModels` 每次调用都通过 `[...spread]` 创建新数组引用
2. `useEffect` 依赖 `models` 数组，引用变化即触发重新执行
3. effect 内的 `dispatch(updateModelStatus(...))` 修改 Redux state → 触发重渲染 → selector 创建新引用 → effect 再次执行
4. 当所有模型测试失败时，`selectedModelId` 始终为 `undefined`，没有任何退出条件

**涉及文件与修复内容**：

| 文件 | 修复方式 |
|------|----------|
| `src/views/DataFormulator.tsx` | 新增 `lastTestedModelKeyRef`，用模型 ID 的排序拼接作为稳定签名。相同签名不重复测试，仅当模型列表真正变化（如 globalModels 异步加载完成）时才重新测试 |
